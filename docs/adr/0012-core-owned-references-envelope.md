# Core-owned references envelope with Spec substitution

References live in the resource envelope as a map keyed by local name — `references: {net: {kind: network, name: backend, field: networkId}}` — a sibling of Spec, written only by the user, where `field` names a key in the target's `Status.Observed`. The **core** resolves them generically: a reference resolves iff the target exists, is Ready, and carries a non-empty value at `field`. When *all* of a resource's references resolve, core substitutes each `$(ref:name)` token inside the Spec's JSON string values on an in-memory copy and hands that substituted Spec to the Provider; the Store always keeps the user's original bytes. When any reference is unresolvable, core skips the Provider entirely for that pass and records `Ready=false`, `Phase=Pending`, and a `waiting for <kind>/<name>: <why>` message, then requeues via the work queue's requeue-after seam — per ADR-0005, an unresolvable Reference is **never an error**, and it never tears down a running Substrate object: a missing dependency makes you unready, not destroyed. Reference cycles need no detector — a cycle is just resources that are mutually unresolvable, so they stall as mutually not-Ready with legible messages, which is the documented behavior. A `$(ref:name)` token naming an undeclared reference *is* an error (400 at PUT time): that is malformed input, not an unready dependency.

Two alternatives were rejected. **Provider-driven resolution** (each Provider decodes reference fields from its own Spec and resolves them) makes every Provider re-implement lookup, readiness-gating, and requeue plumbing — the reference model would live in N places and drift. **A resolved-values map passed alongside the Resource** keeps core in charge of resolution but still leaks reference-awareness into every Provider's decode path; with substitution, Providers stay byte-for-byte ignorant of references (the network provider needed zero changes, and the container provider merely gained an honest `network` Spec field). Keeping the block out of Spec preserves ADR-0004's seal — core still never parses Spec *schema*; token substitution manipulates bytes, not shape.

A deliberate consequence with teeth: the spec-hash fingerprints the **substituted** Spec, so when a referenced resource is replaced (its Observed value changes), dependents' hashes mismatch and the existing drift machinery recreates them onto the new value — replacement cascades re-convergence through the reference graph with no new code path. Accepted costs: dependents restart when a dependency is replaced (correct for network attachment, but a policy Compositions in M3 inherit), and a dangling reference legitimately sits not-Ready forever — failure diagnosis remains "why is this resource stuck unready?", as ADR-0005 promised.
