# GitHub is a Substrate, repo generation is reconciled — not a scaffolder job

Repo generation is modeled as a **GitHub Provider** managing a `Repository` Primitive, not as a Backstage-style one-shot scaffolder workflow. A Repository's Spec declares the repo exists and is bootstrapped with the delivery-process kit (`~/dev/github-agent-roadmap-bootstrap`: labels, issue/PR templates, project board, agent instructions); the reconciler observes GitHub, creates what's missing, and repairs drift (a hand-deleted label gets restored). The bootstrap kit is the Provider's muscle — shelled out to initially, wrapped properly later. An imperative scaffolder subsystem was rejected because it introduces a second, one-shot execution model alongside the reconcilers — the same tension rejected in ADR-0005 — and cannot repair drift.

The dogfood loop this enables: a `Service` Claim expands via Composition into a Repository Primitive (GitHub Provider) plus runtime Primitives (Docker Provider) — one Claim yields a repo with the delivery process installed and running infrastructure. Accepted cost: reconciling against GitHub requires API-rate-limit awareness and slower observe cycles than Docker (cached observations, longer resync intervals).
